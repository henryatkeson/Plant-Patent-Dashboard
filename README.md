# Variety Rights Sourcing Engine

This is a searchable sourcing engine for fruit, tree nut, and vegetable breeding companies, rights managers, nurseries, and variety owners.

## What it tracks

- Seed data from the local workbook in `Patent Dashboard`.
- Public issued plant patents from the USPTO Official Gazette.
- CPVO Variety Finder export records for selected high-relevance crops.
- Published plant applications can be added later through the same JSON file once an Open Data Portal/API-key workflow is selected.

Important limitation: unpublished filings inside Patent Center are not public records. A USPTO account can help with Patent Center or Open Data Portal workflows, but it cannot make non-public filings broadly available for an automated public dashboard.

## Run locally

From this folder:

```powershell
python scripts/export_seed.py
python scripts/refresh_uspto_grants.py --issues 8
python scripts/import_cpvo_varieties.py
python scripts/build_owner_profiles.py
python scripts/serve.py
```

Then open:

```text
http://127.0.0.1:8787/
```

## Daily refresh

The refresh scripts write `data/plant_patents.json` and `data/owner_profiles.json`. `data/cpvo_varieties.json` is rebuilt from local CPVO Variety Finder Excel exports when new exports are downloaded.

The scheduled workflow also rebuilds the internal profile-audit and web-research queues, validates the generated data contracts, and saves `data/refresh_health.json`. A failed validation stops publication before a broken or incomplete dataset is committed.

For an online deployment, schedule:

```powershell
python scripts/refresh_uspto_grants.py --issues 8
```

Run it once per day. The USPTO Official Gazette is weekly, but daily checks are harmless and make the dashboard pick up new issues as soon as they appear.

## CPVO Variety Finder

The dashboard imports CPVO Variety Finder workbook exports with `scripts/import_cpvo_varieties.py`. The importer reads every `*CPOV Varieties.xlsx` workbook saved in the parent `Patent Dashboard` folder, dedupes overlapping records, and writes the combined dataset to `data/cpvo_varieties.json`. Register types are grouped as protected IP (`PBR`, `PLP`), official listings (`NLI`, `FRU`), commercial listings (`COM`), and other/unclear (`ZZZ`).

## Owner Census

The owner/breeder sourcing view is generated by `scripts/build_owner_profiles.py`. It builds sourcing profiles from `data/plant_patents.json` and `data/cpvo_varieties.json`, including crop mix, jurisdiction mix, first/last activity years, five-year filing velocity, estimated expiration buckets, confirmed-assignee counts, annual activity, expiration schedules, and sourcing flags. USPTO records use assignee first when available; CPVO records currently use breeder names from the Variety Finder export because holder/applicant fields are not included in the current Excel downloads. The default dashboard view filters the census to fruit, tree nut, and vegetable signals, while the "All sourcing signals" view keeps ornamentals available for reference.

`recordCount` is the number of source register or jurisdiction observations, not a cultivar count. `distinctCultivarCount` is a normalized count of displayed variety labels and may still differ from the number of legally distinct varieties. Company rollups retain the source records of affiliated profiles and deduplicate overlapping rights by record ID before calculating activity, crop mix, jurisdictions, and expiration cliffs.

Acquisition fit is confidence-gated. A breeder-only individual with no completed web review or confirmed legal-owner record is capped at 50, an unresolved identity is capped at 25, and public institutions or strategic-scale platforms are downranked regardless of portfolio size. A profile can be suppressed from the table only when its records have a configured rollup destination; this prevents a display cleanup from silently removing portfolio observations.

Known company aliases, website links, and short company descriptions live in `config/company_profiles.json`. This file is intentionally explicit so common aliases such as `IFG` / `International Fruit Genetics` can be rolled together without broad fuzzy matching that would merge unrelated breeders.

Source-backed company, affiliation, ownership, contact, and trademark research lives in the versioned `config/profile_web_research*.json` ledgers. Every ledger is loaded during a rebuild, which allows the census to expand in reviewable batches. `scripts/build_web_research_queue.py` gives every generated owner/breeder profile a durable research status and writes `data/web_research_queue.json` plus a spreadsheet-friendly CSV. Search prompts in that queue are research starting points, not evidence; only captured source URLs should affect profile confidence. Public business contact details may be recorded, but private details are not inferred and historical rights are not reassigned to a breeder's newer employer without title evidence.

Run `scripts/validate_dashboard_data.py --strict` after changing company rollups or evidence. It checks that every owner has an audit and research row, company names and aliases are unambiguous, configured URLs are well formed, and the website probe has not detected an unrelated domain.

## Crop filtering

Edit `config/crop_keywords.txt` to change what the Gazette updater treats as fruit, tree nut, or vegetable. The USPTO plant-patent feed includes ornamentals, so this filter keeps the dashboard focused.
